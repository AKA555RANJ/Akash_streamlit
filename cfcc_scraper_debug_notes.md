# Cape Fear Community College — Scraper Debug Notes

**School:** Cape Fear Community College  
**ID:** 3055607  
**Platform:** bkstr.com (Follett) — `svc.bkstr.com` REST API  
**Target URL:** https://www.bkstr.com/capefearstore/shop/textbooks-and-course-materials  

---

## What We Tried

### Attempt 1 — Timber platform URL (wilmingtonstore.cfcc.edu)
- Original bookstore URL `https://wilmingtonstore.cfcc.edu/timber/college` resolves to `216.27.28.88`
- TCP connection times out on port 443 for all non-browser clients
- **Conclusion:** Server is unreachable without a residential/browser proxy. Abandoned in favour of the Follett/bkstr URL.

---

### Attempt 2 — Longwood pattern: FlareSolverr bootstrap + requests.Session
- Cloned `longwood_textbook_scraper.py` (same bkstr.com platform)
- FlareSolverr visits the SPA → gets `_px3` + session cookies → hand off to `requests.Session`
- Added Webshare rotating proxy `p.webshare.io:80` to all FlareSolverr calls
- **Result:** 407 Proxy Authentication Required — sticky session username suffix (`-session-cfcc1`) is not a valid Webshare format; rotating proxy also caused IP mismatch between GET and POST

---

### Attempt 3 — Remove proxy (VPN covers IP routing)
- Removed proxy entirely; used `requests.Session` with FlareSolverr cookies
- GETs to `svc.bkstr.com` (store/config, terms, courses) → **200 OK** ✅
- POST to `svc.bkstr.com/courseMaterial/results` → **403** ❌
- **Root cause:** PerimeterX (PX) applies stricter validation on POST — requires behavioral sensor data (`_pxde`) generated by live JS interaction inside the browser. Cookies extracted from FlareSolverr and replayed in `requests.Session` lack this sensor data.

---

### Attempt 4 — FlareSolverr `session.fetchjson` for POST
- Used `session.fetchjson` (runs `fetch()` inside the Chrome browser tab) for POST
- **First attempt:** HTTP 403 from bkstr — PX still rejected it (sensor data not populated yet on fresh page load)
- **After session refresh:** FlareSolverr returned 500 (internal error after session state reset)
- **Conclusion:** `session.fetchjson` is flaky post-refresh; also doesn't wait for PX sensor data to initialise

---

### Attempt 5 — FlareSolverr `request.post` with `headers` field
- Switched to `request.post` and added `"headers": {"Content-Type": "application/json"}`
- FlareSolverr v3.3.21 does **not** support a `headers` parameter on `request.post`
- **Result:** 500 from FlareSolverr itself
- **Conclusion:** `headers` param is unsupported in this FlareSolverr version

---

### Attempt 6 — `curl_cffi` with Chrome TLS impersonation
- After FlareSolverr bootstrap (PX cookies captured), switched to `curl_cffi.Session(impersonate="chrome120")` for all API calls
- GETs work; POST to `courseMaterial/results` → **403**
- **Root cause confirmed:** Not a TLS fingerprint issue — PX on POST specifically requires live browser sensor data (`_pxde`), which `curl_cffi` cannot generate

---

### Attempt 7 — Playwright with `page.evaluate()` fetch
- Full Playwright rewrite; bootstrap by visiting SPA in headless=False Chrome
- Used `page.evaluate()` to run `fetch()` inside the page for both GET and POST
- **GETs → 403:** `svc.bkstr.com` has CORS restrictions; cross-origin `fetch()` with `credentials: include` from inside `bkstr.com` page is blocked
- **Conclusion:** Cannot use `page.evaluate()` fetch for GET against `svc.bkstr.com`

---

### Attempt 8 — Playwright `page.goto()` for GETs
- Navigated the browser tab directly to each API URL (like FlareSolverr's `request.get`)
- GETs → **403** from `svc.bkstr.com`; navigating back to STORE_HOME timed out
- **Root cause:** Navigating away from the SPA destroys the PX session context; `svc.bkstr.com` then has no valid PX state for the direct GET

---

### Attempt 9 — Playwright `page.request` (APIRequestContext)
- Used `page.request.get()` / `page.request.post()` — Playwright's built-in HTTP client that shares the browser's cookie store but doesn't navigate the page
- GETs → **403** still
- **Root cause:** `page.request` uses Playwright's own HTTP stack (not Chrome's), so the TLS fingerprint differs from the browser. PX on `svc.bkstr.com` rejects it.

---

## Root Cause Summary

`svc.bkstr.com` is protected by **PerimeterX with two tiers:**

| Request type | PX tier | What's needed | Status |
|---|---|---|---|
| GET (store/config, terms, courses) | Medium | Chrome TLS fingerprint + `.bkstr.com` cookies | Works only via Chrome navigation (FlareSolverr `request.get`) |
| POST (courseMaterial/results) | Strict | Chrome TLS + live PX sensor data (`_pxde`) generated by real browser interaction | Blocks all replay approaches |

The core tension:
- **GETs** need Chrome navigating to the URL → cookies carried, TLS OK → works
- **POST** needs the live SPA browser context (sensor data) → only `fetch()` from inside the SPA's Chrome tab works
- `fetch()` from inside `bkstr.com` page to `svc.bkstr.com` → CORS-blocked for GETs
- Chrome navigating away to `svc.bkstr.com` → destroys the SPA PX session needed for POSTs

## Proposed Next Step

**Two-page Playwright approach (not yet run):**
- **Page 1** stays on STORE_HOME (SPA) — used for POST via `page1.evaluate()` (browser fetch, PX intact)
- **Page 2** handles GET navigations to `svc.bkstr.com` — uses Chrome network stack, shares cookies from the same browser context
- Both pages share the same `BrowserContext` so `.bkstr.com` cookies (including `_px3`) are shared
- Page 1's SPA never navigates away, preserving the PX sensor state for POSTs

```python
ctx   = browser.new_context(...)
page1 = ctx.new_page()   # stays on STORE_HOME forever
page2 = ctx.new_page()   # used for GET navigations to svc.bkstr.com

page1.goto(STORE_HOME, wait_until="networkidle")
time.sleep(10)  # let PX fully initialise

# GET: Chrome navigates page2 to API URL
resp = page2.goto("https://svc.bkstr.com/store/config?storeName=capefearstore")
data = json.loads(resp.body())

# POST: SPA fetch() from page1 (PX session intact)
result = page1.evaluate("""
    async ([url, payload]) => {
        const r = await fetch(url, {
            method: 'POST',
            headers: {'Content-Type': 'application/json'},
            body: JSON.stringify(payload)
        });
        return {status: r.status, body: await r.text()};
    }
""", [url, payload])
```

This is the most promising untested approach remaining.
